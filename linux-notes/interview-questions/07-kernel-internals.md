## Q1: Explain the difference between insmod and modprobe. Why should you never use insmod in production?

- `insmod` directly inserts a single `.ko` file using the `init_module()` syscall
- It does NOT resolve dependencies -- if module A depends on module B, `insmod A.ko` fails or causes undefined behavior
- `modprobe` reads `modules.dep` (generated by `depmod -a`) and recursively loads all dependencies first
- `modprobe` reads `/etc/modprobe.d/*.conf` for options, blacklists, aliases, and install/remove hooks
- `modprobe` verifies kernel version compatibility from `/lib/modules/$(uname -r)/`
- `modprobe -r` recursively removes dependent modules (if refcount is 0); `rmmod` does not
- In production, `insmod` risks unresolved symbols, which can cause kernel oops or panic
- Only legitimate `insmod` use: early boot (initramfs) before `/lib/modules` is mounted

## Q2: What does it mean for the kernel to be "tainted"? How do you decode the taint flags and why does it matter?

- The kernel maintains a bitmask at `/proc/sys/kernel/tainted`
- A value of 0 means clean; any nonzero value indicates a taint condition
- Once tainted, the flag persists until reboot -- even if the offending module is removed
- Key taint bits:
  1. Bit 0 (P): Proprietary/non-GPL module loaded (e.g., NVIDIA driver)
  2. Bit 1 (F): Module force-loaded with version mismatch (`--force`)
  3. Bit 2 (S): SMP kernel running on non-SMP-capable hardware
  4. Bit 9 (W): Warning condition occurred (soft lockup, etc.)
  5. Bit 12 (O): Out-of-tree module loaded
  6. Bit 13 (E): Unsigned module loaded when signature enforcement enabled
- Vendor support impact: Red Hat, SUSE, Canonical may refuse to investigate kernel bugs on tainted kernels
- Decode: `python3 -c "t=int(open('/proc/sys/kernel/tainted').read()); [print(f'Bit {i}') for i in range(20) if t & (1<<i)]"`
- Best practice: monitor `/proc/sys/kernel/tainted` fleet-wide and alert on any nonzero value

## Q3: Explain cgroups v2 unified hierarchy. How does it differ from v1, and why does Kubernetes require it?

**cgroups v1:**
- Each controller (cpu, memory, blkio, pids) has its own independent hierarchy
- A process can be in different cgroups in different controller hierarchies
- No consistent cross-controller view of resource usage
- Race conditions when moving processes between cgroups across hierarchies
- Unsafe delegation model (granting unprivileged users cgroup management is dangerous)
- I/O controller (`blkio`) only works with direct I/O, broken for buffered writes

**cgroups v2:**
- Single unified hierarchy at `/sys/fs/cgroup/`
- A process belongs to exactly one cgroup with all enabled controllers
- "No internal processes" rule enforced: leaf cgroups hold processes, internal nodes hold child cgroups
- Safe delegation via `cgroup.delegate` -- unprivileged users can manage subtrees
- PSI (Pressure Stall Information) built-in
- Proper I/O accounting for buffered writes
- `memory.high` for throttling before OOM kill (no v1 equivalent)

**Kubernetes impact:**
- v1.31 deprecated v1 support; v1.35+ refuses kubelet startup on v1 nodes by default
- Memory QoS (using `memory.high`) requires v2
- PSI-based eviction (alpha feature) requires v2
- Container swap support requires v2
- GKE v1.33+ forcibly migrates v1 clusters to v2

## Q4: How would you set a CPU limit of exactly 2.5 cores using cgroups v2? Walk through every step.

1. Enable the CPU controller on the parent:
   - `echo "+cpu" > /sys/fs/cgroup/cgroup.subtree_control`
2. Create a cgroup subtree:
   - `mkdir /sys/fs/cgroup/myapp`
   - `echo "+cpu" > /sys/fs/cgroup/myapp/cgroup.subtree_control`
3. Create the leaf cgroup (v2 requires processes in leaf nodes):
   - `mkdir /sys/fs/cgroup/myapp/worker`
4. Set CPU bandwidth limit:
   - `echo "250000 100000" > /sys/fs/cgroup/myapp/worker/cpu.max`
   - Meaning: 250,000 microseconds of CPU time per 100,000 microsecond period = 2.5 CPUs
5. Move the target process:
   - `echo <PID> > /sys/fs/cgroup/myapp/worker/cgroup.procs`
6. Verify enforcement:
   - `cat /sys/fs/cgroup/myapp/worker/cpu.stat` -- check `nr_throttled` and `throttled_usec` fields
7. systemd equivalent: `systemctl set-property myapp.service CPUQuota=250%`
8. Docker equivalent: `docker run --cpus=2.5 <image>`

## Q5: List all 8 Linux namespace types and give a production scenario for each.

1. **Mount (mnt):** Container gets a private filesystem via `pivot_root()` + overlayfs. Scenario: read-only rootfs with ephemeral tmpfs for `/tmp`, protecting against rootkit persistence.
2. **UTS:** Isolates hostname and NIS domain name. Scenario: Java applications that embed `hostname` in distributed tracing headers need unique hostnames per pod.
3. **IPC:** Isolates System V IPC objects and POSIX message queues. Scenario: preventing a PostgreSQL container's shared memory segments from colliding with another container on the same host.
4. **PID:** Process ID space isolation; first process becomes PID 1. Scenario: `kill -HUP 1` inside a container signals the container's init, not the host's systemd.
5. **Network:** Private network stack (interfaces, routes, iptables, socket space). Scenario: each Kubernetes pod gets its own IP address and routing table via CNI plugin.
6. **User:** Maps container UID 0 to an unprivileged host UID. Scenario: rootless Podman containers where a container escape yields only unprivileged access.
7. **Cgroup:** Container sees its own cgroup as the hierarchy root. Scenario: monitoring agents inside containers read `memory.current` and see their cgroup as `/`, not the host path.
8. **Time:** Per-namespace CLOCK_MONOTONIC and CLOCK_BOOTTIME offsets. Scenario: live-migrating containers between hosts without breaking uptime calculations in monitoring tools.

## Q6: Walk through what happens at the kernel level when a container is OOM-killed.

1. Container process allocates memory, pushing `memory.current` past `memory.max` in its cgroup
2. Page allocator invokes cgroup charge path (`try_charge()` in `mm/memcontrol.c`)
3. Kernel attempts memory reclaim within the cgroup:
   - Reclaims page cache pages
   - Swaps anonymous pages (if `memory.swap.max > 0`)
   - Invokes direct reclaim / kswapd within cgroup scope
4. If reclaim fails to free sufficient memory, cgroup OOM handler invoked (`mem_cgroup_oom()`)
5. OOM killer selects victim process within the cgroup:
   - Score based on `oom_score` (proportional to RSS)
   - Adjusted by `oom_score_adj` (-1000 to +1000; set via `/proc/<pid>/oom_score_adj`)
6. Selected process receives `SIGKILL` (uncatchable)
7. `memory.events` file updates: `oom` and `oom_kill` counters increment
8. Kernel log (`dmesg`): "Memory cgroup out of memory: Killed process <PID> (<comm>)"
9. If `memory.oom.group = 1`, ALL processes in the cgroup are killed simultaneously
10. Container runtime detects non-zero exit status (137 = SIGKILL), reports OOMKilled
11. Orchestrator (Kubernetes) restarts pod or enters CrashLoopBackOff if limit is too low

## Q7: How do you enter a running container's namespaces without Docker CLI?

1. Get the container's init PID on the host:
   - `docker inspect --format '{{.State.Pid}}' <container>`
   - Or: `crictl inspect <container-id> | jq .info.pid`
2. Enter all namespaces: `sudo nsenter -t <PID> --all -- /bin/sh`
3. Selectively enter specific namespaces:
   - `-n` = network only (debug connectivity without affecting mount/PID)
   - `-m` = mount only (inspect files)
   - `-p` = PID namespace
4. For network debugging: `nsenter -t <PID> -n ip addr show`
5. Namespace files are at `/proc/<PID>/ns/` -- compare inodes between host and container
6. When container has no shell: use host binaries via `-r` (root) flag or mount host tools
7. `ip netns exec` only works if namespace is bind-mounted in `/var/run/netns/` (container runtimes typically do not do this)
8. Programmatically: `open("/proc/<PID>/ns/net")` then `setns(fd, CLONE_NEWNET)`

## Q8: Explain the "no internal processes" rule in cgroups v2 and its impact on container runtimes.

- In v2, a non-root cgroup cannot simultaneously hold processes and have child cgroups with controllers enabled
- Processes must reside in leaf cgroups only
- Internal (non-leaf) nodes serve only as hierarchy organizers and controller delegation points
- Rationale: eliminates ambiguity in resource distribution
  - In v1: if a parent cgroup has processes AND child cgroups, how should CPU shares be split?
  - v2 avoids this by making it architecturally impossible
- Exceptions:
  - Root cgroup can hold processes
  - Threaded cgroups (for `cpu` and `cpuset` controllers) relax this rule for per-thread control
- Impact on container runtimes:
  - Must always create leaf cgroup per container, never place processes in intermediate nodes
  - systemd creates `<service>.scope` leaf nodes for each container
  - Code ported from v1 that writes PIDs to non-leaf cgroups will fail with EBUSY on v2

## Q9: What is seccomp-BPF and how does Docker's default seccomp profile work?

- Seccomp (Secure Computing mode) restricts which syscalls a process can invoke
- BPF (Berkeley Packet Filter) provides programmable filter logic applied to each syscall
- Docker's default profile blocks approximately 44 of 300+ syscalls, including:
  - `clone()` with `CLONE_NEWUSER` (no user namespace creation inside container)
  - `mount()`, `umount2()` (no filesystem manipulation)
  - `reboot()` (no host reboot)
  - `kexec_load()` (no kernel replacement)
  - `init_module()`, `finit_module()` (no kernel module loading)
  - `settimeofday()`, `clock_settime()` (no time manipulation)
  - `swapon()`, `swapoff()` (no swap management)
  - `acct()` (no process accounting manipulation)
- Profile is applied after namespace creation but before `execve()` of container entrypoint
- Violation result: returns EPERM by default (or SIGSYS + SIGKILL if profile specifies kill action)
- Custom profiles: `docker run --security-opt seccomp=custom.json`
- Disable entirely: `docker run --security-opt seccomp=unconfined` (dangerous, audit trail recommended)
- Kubernetes: `securityContext.seccompProfile` in PodSpec (GA since v1.19)
- Check process seccomp status: `cat /proc/<pid>/status | grep Seccomp` (0=disabled, 1=strict, 2=filter)

## Q10: How do you debug a container that cannot reach the network? Walk through each layer.

1. **Get container PID:** `docker inspect -f '{{.State.Pid}}' <ctr>`
2. **Enter network namespace:** `nsenter -t <PID> -n bash`
3. **Check interfaces:** `ip link show` -- is eth0 present and UP?
4. **Check IP assignment:** `ip addr show eth0` -- does it have an IP in the expected range?
5. **Check routing:** `ip route show` -- is there a default route pointing to the bridge gateway?
6. **Check DNS:** `nsenter -t <PID> -m cat /etc/resolv.conf` -- correct nameserver?
7. **Test gateway reachability:** `ping <gateway-ip>` from inside namespace
8. **On host -- check veth peer:** `ip link show | grep veth` -- is the peer interface UP?
9. **On host -- check bridge:** `bridge link show` or `ip link show type bridge`
10. **On host -- check iptables:** `iptables -t nat -L -n -v` -- verify MASQUERADE and DNAT rules
11. **Check conntrack:** `conntrack -L | grep <container-ip>` -- are flows being tracked?
12. **Check for DROP rules:** `iptables -L FORWARD -n -v --line-numbers` -- any rules blocking?
13. **Kubernetes-specific:** Check NetworkPolicy via CNI plugin (Calico: `calicoctl get networkpolicy`)
14. **Capture packets:** `nsenter -t <PID> -n tcpdump -i eth0 -nn -c 20` -- traffic leaving container?

## Q11: What are Linux capabilities and why are they critical for container security?

- Capabilities decompose monolithic root privilege (UID 0) into ~41 discrete permission flags
- Key capabilities:
  - `CAP_NET_ADMIN`: configure interfaces, routing, iptables rules
  - `CAP_NET_RAW`: create raw sockets (ping, packet capture)
  - `CAP_NET_BIND_SERVICE`: bind to ports below 1024
  - `CAP_SYS_ADMIN`: mount filesystems, `setns()`, BPF, clone namespaces (practically equivalent to root)
  - `CAP_SYS_PTRACE`: trace any process, read any process memory
  - `CAP_DAC_OVERRIDE`: bypass file read/write permission checks
  - `CAP_CHOWN`: change file ownership arbitrarily
  - `CAP_SETUID/CAP_SETGID`: change process UID/GID
  - `CAP_KILL`: send signals to any process
- Container runtime behavior:
  - Drops all capabilities by default, then adds back a safe subset
  - Docker default set keeps ~14 capabilities (NET_RAW, CHOWN, SETUID, etc.)
- Best practice: `--cap-drop=ALL --cap-add=<only-what-needed>`
- `CAP_SYS_ADMIN` should almost never be granted to containers
- Audit capability usage: `bpftrace -e 'tracepoint:capability:cap_capable { printf("%s CAP_%d\n", comm, args->cap); }'`
- View process capabilities: `getpcaps <pid>` or `cat /proc/<pid>/status | grep Cap` + `capsh --decode=<hex>`

## Q12: Explain cgroup PSI (Pressure Stall Information) and how you would use it for container autoscaling.

- PSI tracks the fraction of time tasks in a cgroup are stalled waiting for a resource
- Three metric files per cgroup: `cpu.pressure`, `memory.pressure`, `io.pressure`
- Each file has two lines:
  - `some` = percentage of time at least one task was stalled
  - `full` = percentage of time ALL tasks were stalled (complete throughput loss)
- Format: `some avg10=X.XX avg60=Y.YY avg300=Z.ZZ total=T`
  - `avg10/60/300` = exponential moving averages over 10s, 60s, 300s windows (0-100)
  - `total` = cumulative stall time in microseconds
- Production use cases:
  1. **Autoscaling:** HPA trigger when `memory.pressure avg60 > 30` -- scale out before OOM
  2. **Pre-OOM detection:** alert when `memory.pressure avg10 > 70` -- upcoming OOM kill
  3. **Noisy neighbor detection:** compare PSI across sibling cgroups under the same parent
  4. **Capacity planning:** weekly trend of `cpu.pressure avg300` to rightsize instances
  5. **SLO enforcement:** correlate `io.pressure full` with application latency percentiles
- Kernel PSI triggers: `poll()` on pressure file with threshold (kernel 5.2+), e.g., "notify when some > 50 for 1 second"
- Kubernetes: PSI-based eviction in alpha (v1.29+), evicts pods when node-level PSI exceeds configured threshold
- Only available in cgroups v2

## Q13: A developer wants to run Docker inside Docker. What are the kernel-level implications and what should you recommend?

**Docker-in-Docker (DinD) kernel requirements:**
1. Inner container must call `clone()` with namespace flags -- needs `CAP_SYS_ADMIN`
2. Inner container must call `mount()` for overlayfs -- needs `CAP_SYS_ADMIN`
3. Typically requires `--privileged` (grants ALL capabilities, disables seccomp, mounts host /dev)
4. Nested cgroup delegation: inner daemon must create child cgroups in its subtree
5. Nested overlayfs requires kernel 5.11+ (previously unsupported)

**Security implications of `--privileged`:**
- Container can mount host filesystems (escape trivially)
- Container can load kernel modules
- Container can access host device files
- Container can manipulate host networking
- Container can ptrace host processes
- Effectively no isolation -- equivalent to root on host

**Recommended alternatives (in order of preference):**
1. **Kaniko / Buildah / buildkit:** Build images without a Docker daemon; no special privileges needed
2. **Docker socket bind mount:** `-v /var/run/docker.sock:/var/run/docker.sock` -- simpler, but inner container controls host Docker
3. **Sysbox runtime:** Safely nests containers using user namespaces without `--privileged`
4. **DinD with rootless Docker:** Inner Docker runs rootless, reducing blast radius

## Q14: How does `memory.high` differ from `memory.max` in cgroups v2? When would you use each?

- `memory.max`: hard limit; exceeding triggers OOM kill after reclaim fails
- `memory.high`: soft limit / throttle point; exceeding triggers aggressive memory reclaim, slowing the process but NOT killing it
- Interaction sequence:
  1. Process allocates memory
  2. Usage exceeds `memory.high` -> kernel puts process in direct reclaim, throttles allocations
  3. If pressure persists and usage exceeds `memory.max` -> OOM kill
- `memory.high` creates a "yellow zone" between normal operation and OOM kill
- Production strategy:
  - Set `memory.high` at 80-90% of `memory.max`
  - Example: `memory.max=2G`, `memory.high=1700M`
  - Monitor `memory.pressure` PSI to detect when throttling is active
  - Java apps can trigger GC in response to allocation slowdown
- Kubernetes integration:
  - `resources.limits.memory` -> maps to `memory.max`
  - Memory QoS feature (v2 only, alpha) automatically sets `memory.high` proportionally
  - Without Memory QoS, only hard OOM kill boundary exists
- v1 comparison: no equivalent of `memory.high`; only `memory.limit_in_bytes` (hard limit)
- `memory.low` (v2): minimum guaranteed memory; protects against reclaim (used for important services)

## Q15: Describe how user namespaces enable rootless containers and their security implications.

- User namespaces remap UID/GID between inside and outside the namespace
- Mapping defined via `/proc/<pid>/uid_map` and `/proc/<pid>/gid_map`
  - Format: `<inside-uid> <outside-uid> <count>`
  - Example: `0 100000 65536` -- container UID 0 maps to host UID 100000
- Inside the namespace: process appears as root (UID 0) with full capabilities in that namespace
- Outside (on host): process runs as unprivileged user (e.g., UID 100000)
- Security benefits:
  1. Container escape yields only unprivileged host access
  2. Files in container rootfs owned by mapped UIDs (e.g., 100000-165535) on host
  3. Cannot overwrite host files owned by real root (UID 0)
  4. Kernel validates capability checks against the user namespace, not the initial namespace
- Rootless runtimes: Podman (default), Docker rootless mode, Singularity, Sysbox
- Limitations:
  - Slight overhead for UID translation on every file access
  - `mknod` for device nodes fails (no privilege in host user NS)
  - NFS and some FUSE filesystems do not support UID remapping
  - Network setup requires `slirp4netns` or `pasta` (no raw socket access without `CAP_NET_RAW` in init NS)
  - Some legacy applications assume they can actually manipulate host resources as root
- `/etc/subuid` and `/etc/subgid` define the range of subordinate IDs available per user

## Q16: Walk through what happens when you run `unshare --pid --fork --mount-proc bash`.

1. **`unshare()` syscall** called with `CLONE_NEWPID`:
   - Kernel creates a new PID namespace
   - The calling process itself stays in the parent PID namespace
   - Only its future children will be members of the new namespace
2. **`--fork`** causes the `unshare` utility to `fork()`:
   - Child process becomes PID 1 in the new PID namespace
   - PID 1 is special: it inherits orphaned processes and must `wait()` to reap them
   - Signals like SIGKILL/SIGSTOP cannot be sent to PID 1 from within the namespace
   - If PID 1 exits, the namespace is destroyed and all processes inside are killed
3. **`--mount-proc`** performs two additional steps:
   - Implicitly creates a new mount namespace (`CLONE_NEWNS`)
   - Mounts a fresh `/proc` that reflects only the new PID namespace
   - Without this, `ps` would read the host's `/proc` and show all host processes
4. **`bash`** is `execve()`'d as the child's process image:
   - bash runs as PID 1 inside the namespace
   - `ps aux` shows only processes spawned within this namespace
   - `echo $$` returns 1
5. **On exit:** when bash (PID 1) exits, the PID namespace is destroyed; all child processes receive SIGKILL
6. **Caveat:** bash as PID 1 does NOT reap zombie children properly -- use `tini` or `dumb-init` in production

## Q17: How would you investigate and fix a situation where containers on a node suddenly cannot be created due to "cannot allocate memory" errors, but the node has plenty of free RAM?

1. **Check PID exhaustion** (most common non-memory cause of "cannot allocate memory"):
   - `sysctl kernel.pid_max` -- default 32768 may be too low for container hosts
   - `ls /proc | grep -c '^[0-9]'` -- count active PIDs
   - Compare with `kernel.pid_max`
2. **Check for zombie accumulation:**
   - `ps aux | awk '$8 ~ /Z/' | wc -l` -- count zombies
   - Zombies hold PID slots but consume no memory
   - Root cause: containers without proper init process (no zombie reaping)
3. **Check cgroup PID limits:**
   - `cat /sys/fs/cgroup/<path>/pids.current` vs `pids.max`
   - Also check parent cgroup limits (limits are hierarchical)
4. **Check file descriptor limits:**
   - `cat /proc/sys/fs/file-nr` -- allocated vs maximum
   - `ulimit -n` for per-process limit
5. **Check inotify limits (container creation needs watches):**
   - `cat /proc/sys/fs/inotify/max_user_instances`
   - `cat /proc/sys/fs/inotify/max_user_watches`
6. **Check cgroup memory limits at the kubelet/system level:**
   - `cat /sys/fs/cgroup/system.slice/memory.current` vs `memory.max`
   - Kubelet's own cgroup may be out of memory
7. **Resolution:**
   - Increase `kernel.pid_max` to 4194304
   - Set `pids.max` on containers (e.g., 512)
   - Require proper init process (tini/dumb-init) in all container images
   - Increase inotify limits in `/etc/sysctl.d/`

## Q18: Explain kernel module signing, Secure Boot integration, and what happens when they conflict.

- **Module signing facility:** kernel compiles with `CONFIG_MODULE_SIG` and optionally `CONFIG_MODULE_SIG_FORCE`
- At build time: kernel generates a keypair; public key embedded in kernel image, private key signs modules
- At load time: kernel verifies module signature against embedded public keys
- **Two enforcement modes:**
  1. Permissive (`CONFIG_MODULE_SIG` without `FORCE`): unsigned modules load but taint kernel (flag 'E')
  2. Strict (`CONFIG_MODULE_SIG_FORCE=y`): unsigned modules rejected with `-EKEYREJECTED`
- **Secure Boot integration:**
  - UEFI Secure Boot validates the bootloader, which validates the kernel
  - Kernel then validates modules via its signing facility
  - Chain of trust: UEFI firmware -> bootloader -> kernel -> modules
- **Machine Owner Key (MOK):**
  - For third-party modules (NVIDIA, ZFS, custom drivers), sign with your own keypair
  - Enroll public key: `sudo mokutil --import mykey.der` (requires reboot + physical console confirmation)
  - Sign module: `kmodsign sha512 private.pem public.der module.ko`
- **Conflict scenario:**
  - Kernel update changes signing key; pre-signed third-party modules now fail verification
  - Resolution: use DKMS with signing hooks to automatically rebuild and re-sign on kernel update
- **Best practice:**
  - Always mirror Secure Boot settings between test and production
  - Automate module signing in CI/CD pipeline
  - Monitor `/proc/sys/kernel/tainted` for signature-related taint flags

## Q19: Compare how Docker, Podman, and LXC use namespaces and cgroups differently.

**Docker:**
- Uses all 6 primary namespaces (mount, UTS, IPC, PID, network, cgroup); user namespace optional
- cgroup driver: `systemd` (recommended) or `cgroupfs`
- Requires root daemon (`dockerd`) -- security concern (daemon has root access)
- Default seccomp profile blocks ~44 syscalls
- Drops capabilities to ~14 by default
- Storage: overlayfs with copy-on-write layers

**Podman:**
- Daemonless architecture -- each container is a direct child process
- Rootless by default: uses user namespace to map container root to unprivileged host UID
- Network namespace with `slirp4netns` or `pasta` (no root for network setup)
- cgroup v2 delegation to unprivileged users (v2's safe delegation model)
- Same OCI runtime (runc) as Docker
- Better systemd integration (containers as systemd user services)

**LXC (Linux Containers):**
- System containers (full OS) vs application containers
- Direct use of kernel APIs (no intermediate daemon)
- Can share namespaces selectively (e.g., share network NS with host)
- More fine-grained namespace control via configuration files
- Used by Proxmox VE for lightweight virtualization
- Foundation that Docker originally built upon (pre-containerd)

**Key difference:** Docker/Podman create application containers (single process focus); LXC creates system containers (full init system, multiple services).

## Q20: A production Kubernetes cluster is being migrated from cgroups v1 to v2. What are the top 5 things that can break?

1. **Monitoring agents reading v1 paths:**
   - v1: `/sys/fs/cgroup/memory/docker/<id>/memory.usage_in_bytes`
   - v2: `/sys/fs/cgroup/system.slice/docker-<id>.scope/memory.current`
   - Impact: metrics go to zero, alerting goes blind
   - Fix: upgrade cAdvisor, node_exporter, Datadog agent to v2-compatible versions

2. **Applications that directly access cgroup filesystem:**
   - Java (pre-JDK 15) reads v1 paths for container-aware memory limits
   - Custom resource monitoring scripts hardcoded to v1 paths
   - Fix: upgrade JDK, update scripts to detect v2 and read correct files

3. **Container runtime cgroup driver mismatch:**
   - If kubelet uses `cgroupfs` driver but systemd manages v2 hierarchy, conflicts arise
   - Both kubelet and containerd/cri-o must use `systemd` cgroup driver on v2
   - Fix: set `--cgroup-driver=systemd` in kubelet config and containerd config

4. **Missing v1-specific features:**
   - `memory.swappiness` per-cgroup not available in v2 (only system-wide)
   - `cpuacct` controller merged into `cpu` -- separate accounting scripts break
   - `blkio` renamed to `io` with different file format
   - Fix: rewrite scripts, test controller behavior differences

5. **Kernel version requirements:**
   - cgroups v2 needs kernel 5.8+ for full feature set (PSI triggers, delegation)
   - Older kernels may have partial v2 support with missing controllers
   - Legacy distros (CentOS 7, RHEL 7, Ubuntu 18.04) default to v1 and may not support v2
   - Fix: ensure kernel 5.8+ minimum; prefer 5.15+ LTS
